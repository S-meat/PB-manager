## UI/UXの方針（汎用報告書アプリと同系統のモノトーン）

依頼者が運用中の「製造部 報告書作成アプリ」（旧5S報告アプリ）とトーンを揃えること。既存の`pb-manager.html`は既にこの方針で実装済みなので、**デザインシステムを維持したまま**改善すること。

### デザイントークン（変更しない）

```css
--border:#e0e0de; --border-strong:#c8c8c5;
--ink:#1a1a18; --sub:#6b6b68; --muted:#9b9b98;

--radius:8px / 12px;
フォント: "Hiragino Sans","Hiragino Kaku Gothic ProN","Noto Sans JP","Yu Gothic"
```

### UI原則
1. **完全モノトーン**。状態の区別は色ではなく濃度差で表現する
   - ★要発注＝黒塗り白抜き ／ まもなく＝グレー塗り ／ OK＝白抜き枠線 ／ 終売・対象外＝破線枠
   - グラフも実績＝黒塗りバー、予測＝破線枠バー
2. **スマホ最優先**：680px以下は下部固定タブバー（アイコン＋ラベル）。入力フォントは16px以上（iOS自動ズーム防止）、タップ領域44px以上
3. **一覧はカード型**（発注案件・マスタ・アラート）。表形式は予測・実績など数値比較が主目的の画面のみとし、先頭列は`position:sticky`で固定
4. **操作は大きなボタンで前進型**：工程更新は「次の工程へ ▶／◀ 戻す」ボタン。小さな数字やアイコンだけのタップ操作は避ける
5. **毎操作にフィードバック**：トースト通知（「出庫を登録しました」等）。破壊的操作はconfirm必須
6. アクセシビリティ：focus-visible、aria-label、prefers-reduced-motion対応を維持

## 厳守事項（変更してはいけないもの）

1. **POSTは `Content-Type: text/plain` のfetch**。application/jsonに変えない（CORSプリフライトでGASが受け取れなくなる）。※報告書アプリで使っている隠しform＋iframe方式は「レスポンス不要な送信」用。本アプリはPOST応答の全状態JSONで画面更新する設計のため、fetch方式を維持すること
2. **スプレッドシートの列順**（技術引き継ぎ書 §3）。GASは列番号で参照している
3. **発注点・予測ロジック**（同 §5）。式を変える場合は依頼者に確認
4. **終売仕様**：発注アラート・発注登録・入庫の対象外、在庫>0の間は出庫可能
5. **モノトーン配色方針**と上記デザイントークン
6. GASデプロイ更新は「デプロイを管理→既存デプロイの新バージョン」（URLを変えない）
7. `ヨコレイ在庫_スプレッドシート投入用.xlsx` の再生成・再変換はしない（P03の棚卸未確定など運用上の状態を含むため）

## Claude Codeでの進め方

1. リポジトリ構成の提案から始める（例：`index.html` / `gas/code.gs` / `docs/` に手順書類。単一HTML構成は維持し、ビルドツールは導入しない）
2. 変更のたびに検証すること：
   - `<script>`内を抽出して `node --check`
   - 主要ロジック（在庫計算・発注点・工程遷移・マスタCRUD・終売除外）はDOMモックのスモークテストを`tests/`に常設化する（既存チャットで使ったテストを再現してよい）
3. UI変更時はスクリーンショット（390px/1200px）で確認
4. コミットは機能単位で日本語メッセージ
5. 不明点は推測で実装せず、依頼者に日本語で確認する
